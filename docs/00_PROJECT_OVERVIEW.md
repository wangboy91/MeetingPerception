# 00 项目总览

## 1. 一句话定义

Meeting Copilot 是一个桌面端实时会议辅助工具：监听电脑会议音频，实时转写，并在会议进行中识别问题、风险、决策点和待办，给用户提供可执行的回应建议。

## 2. 本质理解

它不是录音工具，也不是传统会议纪要工具。

它的本质是：

```text
实时音频流处理系统
+ 会议语义状态机
+ 大模型辅助决策系统
```

从技术上看，它是流式管线：

```text
音频采集 → VAD → ASR → 断句/标点 → 说话人标记 → 会议状态更新 → 触发器 → LLM 建议 → UI 展示
```

从产品上看，它解决的是：

```text
会议中人脑来不及分析、来不及记录、来不及组织回应的问题。
```

## 3. MVP 目标

第一版只做一个本地桌面端产品，优先 Windows。

MVP 必须实现：

1. 启动桌面 App；
2. 选择麦克风输入；
3. 采集系统输出音频；
4. 实时语音活动检测；
5. 实时转文字；
6. 展示会议时间轴；
7. 本地保存会议记录；
8. 基于最近上下文识别问题和风险；
9. 调用大模型给出简短建议；
10. 会后生成简版纪要。

## 4. 第一版不做什么

不要做：

- 多租户 SaaS 后台；
- 团队权限管理；
- Zoom/Teams/飞书官方机器人；
- 浏览器插件；
- 手机端；
- 精准识别远程参会人的真实姓名；
- 复杂知识库；
- 完整 CRM/项目管理系统集成。

## 5. 用户场景

### 场景 A：程序员参加需求评审

用户痛点：

- 产品经理提出需求变更；
- 技术风险没有被及时识别；
- 会后才发现缺少边界条件；
- 不知道如何现场追问。

Copilot 输出：

```text
当前问题本质：这个需求涉及支付链路变更，风险不在页面，而在数据一致性。
建议追问：是否影响老订单、退款流程和异常回滚？
建议回应：可以先拆成灰度范围和回滚方案两个问题确认。
```

### 场景 B：销售参加客户沟通

用户痛点：

- 客户说了很多，但没有抓住真实痛点；
- 不知道客户是否有预算；
- 错过成交信号；
- 会后跟进动作不清楚。

Copilot 输出：

```text
客户痛点：现有流程人工成本高，交付周期长。
成交信号：客户问到了上线周期和服务方式。
建议回应：先确认预算周期和决策流程，再给两档方案。
```

### 场景 C：老板参加项目会议

用户痛点：

- 多人讨论容易跑题；
- 决策没有沉淀；
- 责任人不明确；
- 会后执行断层。

Copilot 输出：

```text
当前会议已形成 2 个决策和 3 个待办。
有一个风险没有责任人：上线前压测范围未确认。
建议现在明确 owner 和截止时间。
```

## 6. 产品差异化

市场上会议工具很多，差异化不要放在“转写准确率”上。

真正差异化应该是：

```text
不同角色的实时辅助建议。
```

可扩展成不同助手：

- 技术评审助手；
- 需求评审助手；
- 销售谈判助手；
- 面试辅助助手；
- 管理会议助手；
- 客户成功助手。

## 7. 成本原则

第一版采用低成本架构：

- 本地采集音频；
- 本地 VAD 过滤静音；
- 本地 ASR 优先；
- 本地 SQLite 存储；
- 只在高价值片段调用 LLM；
- 支持 OpenAI-compatible API，避免绑定单一模型供应商；
- 后续可接 Ollama 做本地建议模型。

## 8. 最小闭环

第一阶段的最小闭环是：

```text
用户点击开始会议
→ App 采集麦克风/系统声音
→ 实时转写出字幕
→ 当有人提出问题时，右侧出现建议
→ 用户结束会议
→ 生成纪要
```

只要这个闭环跑通，就可以开始验证真实价值。
